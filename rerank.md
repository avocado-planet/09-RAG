# RAG における Rerank（リランキング）はなぜ必要か

## 一言で言うと

> **Embedding による検索は「意味が近い」ものを見つけるのは得意だが、「本当にこの質問に答えているか」の判断は苦手。Reranker はその判定を補う。**

---

## まず、Embedding 検索の仕組みを振り返る

Embedding 検索（Bi-Encoder）は以下のように動きます：

```
クエリ  →  [Embedding モデル]  →  ベクトル A
ドキュメント →  [Embedding モデル]  →  ベクトル B （事前に計算済み）

→ A と B のコサイン類似度で順位をつける
```

ここで重要なのは、**クエリとドキュメントは別々にベクトル化される**という点です。お互いを見ずに、それぞれ独立して数値に変換されている。これは図書館の分類番号で本を探すようなもので、「だいたい同じ棚にある」ことは保証するけど、「この質問にピッタリ答えている」かどうかは分からない。

---

## Embedding 検索が失敗する具体例

クエリ: **「Cloud-IAM でバックアップから復元する手順は？」**

Embedding 検索の TOP-4：

| 順位 | 内容 | 問題 |
|---|---|---|
| 1 | 「Cloud-IAM のバックアップは毎日自動で行われます」 | ❌ バックアップの**存在**の説明で、**手順**じゃない |
| 2 | 「バックアップのスケジュール設定について」 | ❌ **設定**の話で、**復元**じゃない |
| 3 | 「災害復旧時のデータ復元手順」 | ✅ これが正解 |
| 4 | 「バックアップのストレージ容量について」 | ❌ 全然違う |

「バックアップ」という単語が含まれるだけで上位に来てしまい、**本当に「復元手順」を説明しているチャンクが3位に埋もれている**。これが Embedding 検索の限界です。

---

## Reranker は何が違うのか

Reranker（Cross-Encoder）は、**クエリとドキュメントを一緒に読んで**判断します：

```
[クエリ + ドキュメント]  →  [Cross-Encoder]  →  関連度スコア（0〜1）
```

人間が「この質問に、この文章は答えてる？」と読み比べるのと同じことをモデルがやっています。

先ほどの例に Reranker を通すと：

| Rerank 後 | スコア | 内容 |
|---|---|---|
| 1 | 0.92 | 「災害復旧時のデータ復元手順」 ✅ |
| 2 | 0.45 | 「Cloud-IAM のバックアップは毎日自動で行われます」 |
| 3 | 0.31 | 「バックアップのスケジュール設定について」 |
| 4 | 0.12 | 「バックアップのストレージ容量について」 |

**「復元手順」を聞いているのだから「復元手順」を書いている文章が1位** — 当たり前のことだけど、Embedding だけではこれができない。

---

## なぜ最初から Reranker だけ使わないのか

Cross-Encoder は精度が高い代わりに**遅い**からです。

| | Embedding (Bi-Encoder) | Reranker (Cross-Encoder) |
|---|---|---|
| 速度 | 高速（ベクトル比較のみ） | 遅い（1ペアずつモデルに通す） |
| 10万件の検索 | ミリ秒 | **数十分〜数時間** |
| 精度 | そこそこ | 高い |
| 事前計算 | できる（インデックス） | できない（毎回計算） |

10万チャンクに全部 Cross-Encoder を通すのは現実的でない。だから**2段構え**にする：

```
10万チャンク
    ↓ Embedding 検索（高速）
TOP-20 に絞る
    ↓ Reranker（精密）
TOP-5 を LLM に渡す
```

Embedding で「だいたい合ってる候補」を素早く絞り、Reranker で「本当に答えているか」を精密に判定する。**速さと精度のいいとこ取り**です。

---

## 体感でどれくらい違うのか

RAG の精度改善で **最もコスパが良いのが Reranker の導入** と言われます。

よくある改善の効果の比較イメージ：

| 改善策 | 精度向上の度合い | コスト |
|---|---|---|
| Embedding モデルを `small` → `large` に変更 | 小〜中 | Embedding 再計算 + API コスト増 |
| チャンクサイズの調整 | 中 | インデックス再構築 |
| **Reranker の追加** ⭐ | **大** | API コスト微増（20件×数円程度） |
| ハイブリッド検索 (BM25 + Vector) | 中〜大 | ほぼゼロ |

Reranker は既存の検索パイプラインに **1行追加するだけ** で導入でき、インデックスの再構築も不要。それでいて精度向上が最も体感しやすい。

---

## まとめ

Rerank が必要な理由は結局シンプルです：

**Embedding 検索は「似た話題のチャンク」は見つけられるが、「質問に答えているチャンク」を見つけるのは別の能力。Reranker がその能力を持っている。**

実務で RAG を作るときの鉄板構成は：

```
Hybrid 検索 (BM25 + Vector) で TOP-20
    ↓
Reranker で TOP-5 に精査
    ↓
LLM で回答生成
```

これが現在の RAG のデファクトスタンダードです。
